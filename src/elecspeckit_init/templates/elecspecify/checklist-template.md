# 规格质量检查清单: [硬件特性名称]

**目的**: 在进入 `/elecspeckit.plan`、`/elecspeckit.tasks`、`/elecspeckit.docs` 之前，检查 `spec.md` / `plan.md` 的完整性与可测试性。  
**创建日期**: [DATE]  
**特性**: `specs/[###-feature-name]/spec.md`  

> 此模板由 `/elecspeckit.checklist` 命令参考使用，生成 `FEATURE_DIR/checklists/requirements.md` 等检查清单。  

## 内容质量

- [ ] 规格聚焦硬件用户价值与业务需求，而不是直接堆砌器件型号或实现细节  
- [ ] 规格用中文表述清晰，关键技术术语使用英文，避免混淆  
- [ ] 所有用户故事都用“给定/当/则”结构描述验收场景  
- [ ] 规格中没有明显互相矛盾的描述（例如同时要求“极低成本”和“过度冗余”但未解释折中方式）  

## 需求完整性

- [ ] 所有功能需求（FR）均可测试且有明确成功条件  
- [ ] 非功能需求（如性能、可靠性、功耗、环境等）有可量化指标（SC），而不是“高/低/快/慢”这类模糊词  
- [ ] 边缘情况章节列出了关键边界工况（温度、电压、负载、启动/掉电场景等）  
- [ ] 规格中没有残留 `[NEEDS CLARIFICATION]` 或类似占位标记，所有高优先级模糊点已通过 Clarifications 区域得到解释  
- [ ] 与项目宪法（`.elecspecify/memory/constitution.md`）中 MUST 级原则没有明显冲突  

## 特性就绪度

- [ ] 对每个用户故事都能给出“独立测试”的方法和环境约束  
- [ ] 规格足以指导板级架构（层叠/接口/器件）和验证策略的设计，不需要大量“口头补充”  
- [ ] 规格中关键决策（例如取舍成本 vs 冗余 vs 性能）有明确记录，便于后续追踪和回顾  
- [ ] 规格已准备好作为 `/elecspeckit.plan` 的输入，不需要在 plan 阶段再大量补写基本信息  

## 注意事项

- 完成检查后，将已满足的条目前的 `[ ]` 改为 `[x]`；  
- 如发现严重缺口，应回到 `spec.md` 并通过 `/elecspeckit.clarify` 或手工编辑补齐，而不是在实现阶段“边做边改”；  
- 如本清单中存在不适用的条目，可在本特性的上下文中删除或备注原因。
